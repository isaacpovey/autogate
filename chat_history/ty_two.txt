  Autogate dashboard — implementation design

  1. Scope (confirmed)

  evidence, reason-gated override/block, rollback, confirm modal, timeline) ·
  Stats/Trust. Skipping Settings + My runs.

  2. Architecture

  - Route-based nav (my recommendation, replacing the prototype's in-memory view
  state): / = Open reviews, /runs/[runId] = Run detail, /stats = Trust. Each
  page is an RSC that prefetches its query then hydrates a client view with
  useSuspenseQuery — the exact pattern already in app/page.tsx. Wins: shareable
  run URLs, working back button, server prefetch. The rail/topbar live in a
  client AppShell in the dashboard layout.
  - Live freshness = polling + invalidation (no SSE exists yet; root.ts confirms
  it's deferred). runs.list and the open runs.byId get a refetchInterval;
  override/rollback mutations invalidate. The topbar live/reconnecting dot
  reflects real query fetch/error status — not a faked blip.
  - lib/view-model.ts: pure, unit-tested functions deriving everything the
  prototype computed client-side (riskBand, effortBand, disagreement,
  layerStance, grouping, incident detection, recorded-reason extraction) plus
  the verdict/decision/status style maps. This is the TDD core.
