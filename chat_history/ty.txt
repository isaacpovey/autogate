  1. Package & app layout

  Three pieces, following the spec's "contracts is the single shared dependency"
  principle:

  Where: packages/contracts
  Role: Zod schemas + inferred TS types (Verdict, Finding, RunSummary, …). Pure
    data contracts. No tRPC, no runtime. (ticket 00)
  Depends on: —
  ────────────────────────────────────────
  Where: packages/api
  Role: The tRPC layer: initTRPC setup, Context type, router/publicProcedure,
  the
    appRouter with procedures, and export type AppRouter. The shared API
  contract
     both ends import.
  Depends on: contracts
  ────────────────────────────────────────
  Where: apps/api
  Role: Orchestrator/API app. Wires in-memory mock deps into a context and
  serves
     appRouter over HTTP via tRPC's standalone adapter (pure Node, no framework
  —
     this is what replaces Hono). Port 3002.
  Depends on: packages/api
  ────────────────────────────────────────
  Where: apps/web
  Role: Dashboard. Adds a typed tRPC client bound to AppRouter + one page making

    a real typed call.
  Depends on: packages/api (type-only), contracts

  The key win: apps/web imports type AppRouter from packages/api and gets
  end-to-end type inference — rename a field in a procedure and the dashboard
  fails to compile.

  Why a separate packages/api rather than putting the router in apps/api or in 
  contracts: the dashboard should depend on a package, not another app (clean
  app→package graph), and the router (schema + resolver) is runtime code that
  doesn't belong in the pure-data contracts package. This matches the
  create-t3-turbo layout.

  Does this layout look right?

  2. The scaffold's example procedures

  Two procedures, enough to prove the wiring and set conventions without porting
  the full §6 surface:

  - health — query, no input → { status: 'ok', time: string }. Proves the
  transport works.
  - runs.list — query, Zod input { repo?, status?, limit? } → RunSummary[]
  pulled from an in-memory mock in the context. Proves Zod-validated I/O +
  shared contracts types flowing server→client, and seeds the real DashboardApi
  shape (runs router).

  Resolvers read their dependencies off ctx (functional (deps) => (args) style
  from the spec) — createContext injects the in-mem mock store, so swapping in
  the real Postgres adapter later needs no procedure changes.

  3. Transport & runtime

  - Server adapter: @trpc/server/adapters/standalone (createHTTPServer) —
  zero-framework Node HTTP server, with CORS allowing the dashboard origin. This
  is the direct Hono replacement.
  - Client: @trpc/client + @trpc/react-query + @tanstack/react-query v5,
  httpBatchLink → http://localhost:3002. A 'use client' provider in the Next 16
  App Router, and one server-rendered-then-hydrated page calling health +
  runs.list.
  - tRPC v11 throughout (React 19 / Next 16 compatible).
  - SSE streams (spec §6 /api/stream): map cleanly to tRPC v11 SSE subscriptions
  — I'll note this in the docs but not implement it in the foundation.

  4. Doc changes (the "dump Hono" part)

  - Tickets 00-contracts, 01-orchestrator, 11-dashboard: updated to reference
  the tRPC router / AppRouter / typed client instead of REST + mock server.

⏺ Fetched the docs. Here's the honest answer.

  Why I built it standalone: the Autogate design describes a separate
  long-running orchestrator backend — a Postgres-backed job queue with workers,
  a Docker sandbox that clones PRs, fan-out to Claude agents. None of that is a
  Next.js workload (you can't run a persistent queue worker or spawn Docker from
  a route handler). The design literally says "Orchestrator (Node + tRPC)", so
  I put the dashboard's API on that service and made the dashboard a thin HTTP
  client. That's an explicitly supported tRPC pattern — the same docs say "if 
  your router is on a separate server, pass a client instead", which is exactly
  what my server.tsx does.

  But you're right that it's the wrong default here. The documented, idiomatic
  App Router setup is the route handler at app/api/trpc/[trpc]/route.ts (fetch
  adapter), and for a foundation scaffold it's strictly better:
  - One process, no CORS, no second server to boot.
  - RSC calls procedures directly in-process (no HTTP hop) — so next build even
  prerenders without anything else running. My version needed the standalone
  server up to build.
  - It's literally the docs you linked.

  I made a defensible call but should have surfaced it as a fork instead of
  silently picking standalone — that's on me.

  The clean fix keeps the good part and drops the wart: packages/api stays (the
  router/contract is still a shared package the future worker + CLI can import),
  I delete the standalone apps/api, and mount appRouter in a Next route
  handler. The orchestrator (ticket 01) becomes a headless worker that shares
  the Postgres Store — the dashboard's override/rollback mutations enqueue jobs
  the worker picks up, so Next never needs Docker/agents. That separation is
  actually cleaner than what I had.

  What /goal actually is (and isn't)

  It's not a file — it's a session command: /goal <condition>. You give it one
  completion condition (≤4,000 chars) and Claude keeps taking turns toward it

  conversation transcript. It cannot run commands or read files. So the
  condition must be phrased as something Claude's own surfaced output proves.
  "pnpm turbo check-types passes" works only because Claude runs it and the

  everything we write.
